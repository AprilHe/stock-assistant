# Stock Assistant Decision Engine
## Implementation Plan

> Status: canonical planning document
> Supersedes: `planning/v1.md`
> Audience: product, architecture, and implementation work
> Last updated: 2026-03-15

---

## 1. Goal

Build a trading decision engine that improves proposal quality by separating:
- deterministic signal generation
- signal synthesis and context interpretation
- portfolio fit and risk checks
- execution planning
- final human-readable recommendation output

The system is for **decision support**, not autonomous trading.

---

## 2. Product Scope

### 2.1 What we are building
A multi-strategy research and recommendation system that can:
- run deterministic strategies on a ticker universe
- aggregate and compare strategy outputs
- apply user-specific portfolio constraints
- produce portfolio-aware proposed actions
- generate a clean proposal for Telegram, web, CLI, and reports

### 2.2 What we are not building in v1 of this architecture
- direct broker execution
- fully autonomous PM decisions without human review
- intraday high-frequency systems
- large-scale live optimization loops
- online backtests during each decision request

---

## 3. Core Design Rules

### 3.1 Deterministic first, AI second
Use code for anything that can be computed reliably.

Deterministic layers should own:
- price and indicator calculations
- strategy scores
- normalization
- ranking
- portfolio constraints
- sizing rules
- execution templates
- backtest metrics

AI layers should own:
- news interpretation
- conflicting-signal summaries
- memo generation
- explanation and communication
- natural-language preference parsing if needed

### 3.2 One responsibility per layer
- Strategy layer: generate comparable signals
- Synthesis layer: explain agreement, conflict, and context
- Portfolio layer: decide fit, rank, and size
- Execution layer: define entry/add/stop logic
- Final proposal layer: present the decision clearly

### 3.3 Auditability is required
Every recommendation must be traceable to:
- raw market inputs
- strategy outputs
- normalization assumptions
- portfolio constraints applied
- sizing logic
- execution logic
- generated summary text

1. Data ingestion
2. Strategy signal generation
3. Signal normalization and candidate merge
4. Signal synthesis
5. User profile and constraint application

7. Execution planning
8. Final proposal generation

## 4.1 Data Window Policy

Historical data windows must be defined by use case rather than using one universal lookback.

### Signal calculation windows
- short-term breakout and pullback: `3mo` to `12mo` daily bars
- mean reversion: at least `6mo` daily bars
- volatility and ATR-style features: `20d`, `60d`, and `120d`
- relative strength and trend state: `3mo`, `6mo`, and `12mo`

### Context windows
- market regime detection: `6mo` to `24mo`
- sector and benchmark confirmation: `3mo` to `12mo`
- news and event interpretation: last `3d` to `14d`
- earnings or event freshness: last completed event plus next scheduled event when available

### User-facing display windows
- short summary performance: `20d`
- proposal context: `1mo` and `3mo`
- optional research detail: `6mo` and `12mo`

### Backtesting windows
- minimum useful range: `3y`
- preferred range for strategy evaluation: `5y` to `10y`
- when possible, include both bull, bear, and sideways regimes

### Implementation rule
Each strategy config should declare:
- `signal_lookback_window`
- `feature_windows`
- `min_history_required`

The engine should reject or down-rank a signal if the minimum required history is unavailable.

## 5. Layer-by-Layer Implementation

## 5.1 Layer 1: Data Ingestion

### Responsibilities
Collect all raw inputs required by downstream layers.

### Inputs
- OHLCV price data
- benchmark and sector ETF data
- macro proxies where needed
- recent news headlines
- earnings or event metadata when available
- user holdings
- cash and account size
- user preference profile

### Historical coverage requirements
- live signal generation: enough data for each strategy's declared `min_history_required`
- proposal context: at least `3mo` for chart and trend framing
- backtest-ready research mode: at least `3y`, preferably `5y+`

## 5.2 Layer 2: Strategy Signal Generation

### Responsibilities
Run deterministic strategies and produce a shared signal schema.

### Required signal schema
```json
{
  "ticker": "XOM",
  "strategy_id": "breakout",
  "direction": "long",
  "score_raw": 82,
  "score_normalized": 0.68,
  "confidence": 0.64,
  "horizon": "3-10d",
  "validity_template": "expires_after_3_trading_days_or_on_breakout_failure",
  "entry_logic": "close above 20-day high with volume confirmation",
  "exit_logic": "close below 20-day high or trailing stop violation",
  "risk_flags": ["weak_sector_confirmation"],
  "evidence": [
    "close above 20-day high",
    "volume ratio above 1.5x"
  ],
  "metadata": {
    "atr_pct_14d": 3.4,
    "volume_ratio": 1.8
  }
}
```

### Rules
- Each strategy must emit the same top-level fields
- `score_raw` stays strategy-specific
- `score_normalized` must be comparable across strategies
- `confidence` expresses signal reliability, not expected return
- `horizon` should be explicit and finite
- each strategy must define its expected proposal validity template

### Normalization approach for first implementation
- keep existing per-strategy scoring logic
- add a calibration map from offline backtests
- define `score_normalized` as a 0-1 value after calibration
- if no calibration exists yet, use a documented heuristic fallback

### Proposed module changes
- keep `research/strategies/*.py` as strategy implementations
- add shared schema in `domain/schemas/signals.py`
- add strategy adapter/loader in `app/services/signal_service.py`

### API candidate
- `POST /api/signals/run`

### Request
```json
{
  "tickers": ["AAPL", "XOM", "NVDA"],
  "strategies": ["breakout", "pullback", "mean_reversion"],
  "asset_type": "stock"
}
```

### Response
```json
{
  "generated_at": "2026-03-15T09:30:00Z",
  "signals": []
}
```

## 5.3 Layer 3: Signal Normalization and Candidate Merge

### Responsibilities
Convert multiple strategy signals into ticker-level candidate state before any AI summary runs.

### Questions answered
- which strategies are positive for this ticker
- which strategies conflict
- what is the aggregate directional bias
- which names deserve portfolio evaluation

### Output schema
```json
{
  "ticker": "XOM",
  "aggregate_direction": "long",
  "aggregate_score": 0.63,
  "aggregate_confidence": 0.66,
  "strategy_signals": [
    {"strategy_id": "breakout", "score_normalized": 0.68, "direction": "long"},
    {"strategy_id": "momentum", "score_normalized": 0.55, "direction": "long"},
    {"strategy_id": "mean_reversion", "score_normalized": 0.10, "direction": "flat"}
  ],
  "agreement_level": "medium",
  "conflicts": ["mean_reversion not aligned with trend signals"],
  "preliminary_rank": 4
}
```

### First-pass merge logic
- group by ticker
- weight strategy signals by offline calibration weight
- penalize direct conflict
- require minimum aggregate score to enter candidate set
- keep reasons for every uplift and penalty

### Proposed module
- `app/services/candidate_merge_service.py`

### API candidate
- internal service first, optional external endpoint later

## 5.4 Layer 4: Signal Synthesis

### Responsibilities
Use AI to explain the merged candidate state using market context and news.

### Input contract
- aggregated candidate data
- underlying strategy evidence
- market tone summary
- recent headlines

### Output schema
```json
{
  "ticker": "XOM",
  "overall_direction": "long",
  "conviction": "medium",
  "signal_alignment": {
    "breakout": 0.68,
    "momentum": 0.55,
    "mean_reversion": 0.10
  },
  "summary": "Trend and breakout signals are aligned, but weak sector breadth caps conviction.",
  "key_supports": [
    "trend and breakout agree",
    "defensive sector characteristics"
  ],
  "key_risks": [
    "sector breadth is mixed",
    "setup fails if breakout support breaks"
  ]
}
```

### Important boundary
This layer does not decide:
- position size
- eligibility
- portfolio fit
- final action

### Proposed module
- `core/signal_synthesis.py`

### API candidate
- internal service for now

## 5.5 Layer 5: User Profile and Constraint Engine

### Responsibilities
Convert user and account information into deterministic portfolio constraints.

### Canonical profile schema
```json
{
  "profile_id": "default",
  "risk_profile": "moderate",
  "target_invested_ratio": 0.7,
  "max_single_position": 0.08,
  "max_new_position_size": 0.03,
  "max_sector_exposure": 0.25,
  "max_weekly_turnover": 0.3,
  "allow_shorting": false,
  "allow_options": false,
  "preferred_horizon": "swing",
  "restricted_tickers": [],
  "preferred_sectors": ["Energy"],
  "avoid_sectors": []
}
```

### Canonical portfolio schema
```json
{
  "cash": 25000,
  "equity_value": 100000,
  "current_invested_ratio": 0.62,
  "holdings": [
    {"ticker": "MSFT", "weight": 0.07, "sector": "Technology", "beta_bucket": "high"}
  ],
  "sector_exposure": {
    "Technology": 0.23,
    "Energy": 0.06
  }
}
```

### Deterministic rules
- reject restricted tickers
- reject disallowed instrument types
- reject if max sector exposure would be exceeded
- reduce size if current invested ratio is already near limit
- reduce size for correlated concentration
- reject if horizon conflicts with user preference

### Proposed module
- `core/portfolio_constraints.py`

### API candidate
- `GET /api/profile`
- `PUT /api/profile`
- internal portfolio snapshot service first

## 5.6 Layer 6: Portfolio Construction and Sizing

### Responsibilities
Turn eligible candidates into portfolio-level proposed actions.

### Output schema
```json
{
  "ticker": "XOM",
  "eligible": true,
  "action": "initiate_long",
  "target_weight": 0.03,
  "sizing_method": "starter_size_from_profile_and_signal_strength",
  "rank_within_candidates": 4,
  "portfolio_rationale": [
    "fits sector limits",
    "adds non-tech exposure",
    "moderate conviction supports starter size"
  ],
  "constraints_applied": [
    "max new position size = 3%",
    "current invested ratio below target"
  ],
  "rejection_reasons": []
}
```

### First implementation rules
- start with long-only support
- cap new position by profile limits
- reduce size for high volatility and concentration
- prioritize top-ranked candidates when cash is limited
- support explicit rejection reasons for transparency
- use strategy horizon and proposal validity when selecting action urgency

### Proposed module
- `app/services/portfolio_engine.py`

### API candidate
- `POST /api/proposals/build`

## 5.7 Layer 7: Execution Planner

### Responsibilities
Translate portfolio actions into actionable but non-broker-bound plans.

### Output schema
```json
{
  "ticker": "XOM",
  "entry_style": "starter_position",
  "entry_condition": "enter while price remains above 20-day breakout level",
  "add_condition": "add only after two closes above support",
  "stop_condition": "exit if breakout fails back below the trigger zone",
  "take_profit_condition": "trim if extension exceeds predefined ATR threshold",
  "execution_notes": [
    "do not chase large opening gaps",
    "use smaller initial size in elevated volatility"
  ]
}
```

### First implementation design
Use deterministic execution templates by strategy family:
- breakout family
- pullback family
- mean-reversion family
- commodity macro family

### Validity policy
Execution planning must produce proposal timing metadata, not just entry rules.

Required fields:
- `issued_at`
- `valid_until`
- `review_after`
- `invalidate_if`

Example:
```json
{
  "ticker": "XOM",
  "issued_at": "2026-03-15T09:30:00Z",
  "valid_until": "2026-03-18T21:00:00Z",
  "review_after": "2026-03-17T21:00:00Z",
  "invalidate_if": [
    "price closes below breakout level",
    "opening gap exceeds chase threshold"
  ]
}
```

### Suggested default validity rules
- breakout setup: valid for `1` to `3` trading days unless invalidated earlier
- pullback continuation: valid until support is lost or `2` to `5` trading days pass
- mean reversion: valid for `1` to `3` trading days due to faster decay
- commodity macro: valid for `3` to `10` trading days with event-sensitive review points

### Proposed module
- `core/execution_planner.py`

## 5.8 Layer 8: Final Proposal Generation

### Responsibilities
Create the final user-facing recommendation object and channel-specific rendering.

### Canonical proposal schema
```json
{
  "generated_at": "2026-03-15T09:30:00Z",
  "profile_id": "default",
  "market_context": {
    "regime": "mixed",
    "risk_tone": "neutral"
  },
  "proposed_actions": [
    {
      "ticker": "XOM",
      "action": "initiate_long",
      "target_weight": 0.03,
      "issued_at": "2026-03-15T09:30:00Z",
      "valid_until": "2026-03-18T21:00:00Z",
      "conviction": "medium",
      "reason": "Constructive breakout with acceptable portfolio fit and diversification benefit.",
      "signal_summary": {
        "aggregate_score": 0.63,
        "agreement_level": "medium"
      },
      "execution_plan": {
        "entry_style": "starter_position",
        "entry_condition": "enter while price remains above breakout level"
      }
    }
  ],
  "portfolio_notes": [
    "No additional technology exposure allowed today.",
    "Total invested ratio remains within target."
  ],
  "risk_summary": [
    "Breakout must hold above support.",
    "Sector confirmation remains mixed."
  ]
}
```

### Guidance
- the final proposal can use LLM summarization
- the final action list must come from deterministic upstream outputs
- if LLM fails, fall back to a template renderer
- the final proposal must display validity clearly in every channel
- expired proposals should render as stale rather than active
- actionable ideas should prefer entry ranges over a single exact price
- entry price logic should come from deterministic execution rules, not only from LLM wording

### Proposed modules
- `app/services/proposal_service.py`
- `app/renderers/proposal_renderer.py`

### API candidate
- `GET /api/proposals/latest`
- `POST /api/proposals/render`

## 5.9 Report Output Policy

Daily reports should not only say `Buy`, `Hold`, or `Sell`. They should provide enough structure for a user to act or consciously decide not to act.

### Report types
1. watchlist summary
2. market overview
3. featured idea

### Common rule for actionable recommendations
If a report recommends a trade idea, it should include:
- `action`
- `entry_range`
- `ideal_entry`
- `valid_until`
- `invalidate_if`
- `why_now`

If the engine does not have enough signal quality, it should explicitly output:
- `no_actionable_idea`

### Price recommendation rule
- prefer an entry range instead of one exact price
- allow `ideal_entry` as a reference point within that range
- derive `entry_range` from deterministic execution logic such as breakout level, pullback support, ATR band, or mean-reversion zone
- if price is too extended, the report should say `wait` rather than forcing an entry price

### 5.9.1 Watchlist Summary

The watchlist section is for user-selected names and should provide per-ticker proposals rather than only a rank.

### Required fields per ticker
- `ticker`
- `action`
- `confidence`
- `entry_range`
- `ideal_entry`
- `take_profit_zone`
- `stop_loss_or_invalidation`
- `valid_until`
- `why_now`

### Example schema
```json
{
  "type": "watchlist_summary",
  "generated_at": "2026-03-15T09:30:00Z",
  "items": [
    {
      "ticker": "AAPL",
      "action": "buy",
      "confidence": "medium",
      "entry_range": "208.0-211.5",
      "ideal_entry": 209.4,
      "take_profit_zone": "219.0-223.0",
      "stop_loss_or_invalidation": "close below 205.0",
      "valid_until": "2026-03-18T21:00:00Z",
      "why_now": "pullback into support while trend structure remains intact"
    }
  ]
}
```

### 5.9.2 Market Overview

The market overview should describe the tape and may optionally surface one featured sector or one featured stock.

### Selection rule
- if market leadership is broad and sector rotation is clear, prefer a featured sector
- if no clear sector stands out but one name has unusually strong setup quality, allow a featured stock
- if neither reaches the quality threshold, output `no featured idea today`

### Market overview schema
```json
{
  "type": "market_overview",
  "generated_at": "2026-03-15T09:30:00Z",
  "risk_tone": "neutral",
  "summary": "Breadth is mixed and leadership is concentrated in defensives.",
  "featured_sector": {
    "sector_name": "Energy",
    "reason": "relative strength and supportive commodity backdrop",
    "representative_tickers": ["XLE", "XOM"],
    "entry_style": "buy pullbacks above support",
    "valid_until": "2026-03-19T21:00:00Z"
  },
  "featured_stock": null
}
```

### Featured stock rule
If a single stock is recommended from market overview, it must include:
- `ticker`
- `entry_range`
- `ideal_entry`
- `valid_until`
- `invalidate_if`

### 5.9.3 Featured Idea

The featured idea is the single highest-priority actionable idea of the day. It can be a sector or a stock, but not both at equal priority.

### Stock featured idea schema
```json
{
  "type": "featured_idea",
  "idea_kind": "stock",
  "generated_at": "2026-03-15T09:30:00Z",
  "ticker": "XOM",
  "action": "buy",
  "entry_range": "118.5-120.0",
  "ideal_entry": 119.1,
  "valid_until": "2026-03-18T21:00:00Z",
  "invalidate_if": [
    "close below breakout support",
    "opening gap above 123.0"
  ],
  "why_now": "aligned breakout, supportive macro backdrop, acceptable portfolio fit"
}
```

### No-idea schema
```json
{
  "type": "featured_idea",
  "idea_kind": "none",
  "generated_at": "2026-03-15T09:30:00Z",
  "status": "no_featured_idea_today",
  "reason": "signal quality is mixed and no setup passed the minimum threshold"
}
```

### Proposed modules
- `app/services/report_service.py`
- `app/renderers/proposal_renderer.py`

### Delivery rule
- Telegram should show compact action, price range, and validity
- web can show the full rationale, invalidation logic, and backtest context
- CLI and markdown reports should default to concise tables plus one-line rationale

## 6. Offline Research Layer

### Purpose
Support calibration and monitoring without being part of the live decision path.

### Outputs needed by online system
- strategy calibration weights
- regime-specific reliability adjustments
- confidence adjustment tables
- turnover and drawdown reference metrics

### User-visible backtest metrics
Backtesting should expose both return and risk so users can judge whether a strategy is attractive enough for live use.

Required metrics:
- total return
- annualized return
- annualized volatility
- max drawdown
- Sharpe ratio
- win rate
- exposure
- trade count

Recommended metrics:
- Sortino ratio
- profit factor
- average holding period
- turnover
- benchmark-relative return
- alpha versus benchmark
- rolling 12-month hit rate

### Presentation rule
- research view: show full metric table plus equity curve and drawdown curve
- decision view: show only the key metrics needed for confidence calibration
- do not overwhelm Telegram or short reports with full research output

### Backtest result schema target
```json
{
  "ticker": "XOM",
  "strategy": "breakout",
  "start_date": "2021-01-01",
  "end_date": "2026-01-01",
  "metrics": {
    "total_return_pct": 42.1,
    "annualized_return_pct": 7.3,
    "annualized_volatility_pct": 18.4,
    "max_drawdown_pct": 12.9,
    "sharpe_ratio": 0.71,
    "sortino_ratio": 1.03,
    "win_rate_pct": 48.0,
    "exposure_pct": 52.0,
    "turnover_pct": 88.0,
    "benchmark_return_pct": 31.2,
    "alpha_vs_benchmark_pct": 10.9
  }
}
```

### Suggested artifacts
- `data/calibration/strategy_weights.json`
- `data/calibration/regime_adjustments.json`
- `reports/research/*.md`

### Important rule
Backtesting informs live weights. It does not directly generate live recommendations.

## 11. Decision Rules for Future Changes

If a new feature is proposed, ask:
- does it belong to signal generation, synthesis, portfolio, execution, or presentation
- can it be deterministic
- does it change schemas or only rendering
- does it improve proposal quality or only add complexity

If the answer is only “it sounds smarter,” it should probably not be added yet.
